# Дневник прототипирования

Здесь — не «что сделано» (это в CHANGELOG), а **как думали**: гипотезы, развороты, что осознанно резали. Цель проекта — освоить цикл быстрого прототипирования: *намерение → промпт → живой прототип → ревью → следующая итерация*.

---

## 2026-06-13 · Учебный пример: артефакт намерения

Разобрали первый шаг цикла на нейтральном примере — роль-админка. Главный урок: **прототип начинается не с промпта, а с артефакта намерения** — что именно и зачем проверяю, какой сценарий прохожу в превью, по какому критерию пойму, что прототип удался, и что осознанно режу.

Результат: `docs/spec-admin-panel-v1.md`.

## 2026-06-15 · Прототип v0.1 — «рабочий стол для себя»

Гипотеза: даже без работы за деньги жизнь состоит из потоков (расписание, поиск работы, проекты, спортзал, еда, уборка, растяжка) — соберу «админку для себя».

Решения по ходу:
- Формат — **живой артефакт**, а не статичный файл: подтягивает реальные данные из Google Calendar при каждом открытии. Это и есть «взрослый» способ прототипировать — поверхность поверх реальных данных.
- Перед сборкой **прощупал API календаря** (`list_events`), чтобы строить парсер под реальную форму ответа, а не догадки. Нашёл: события приходят с `start.dateTime` либо `start.date` (события на весь день), плюс `location`, `conferenceUrl`.
- Данные — смешанные: расписание живое, рутины/работа/проекты — руками, с сохранением в localStorage.

## 2026-06-15 · Разворот → v0.2 «движок задач»

Ключевой инсайт во время ревью первого прототипа: пассивная витрина скучна. **Развернули концепцию** — рабочий стол как очередь задач, куда задачи *прилетают*: либо я добавляю, либо генерят доменные AI-агенты, и часть улетает реальными событиями в календарь.

Развилки, которые зафиксировали перед сборкой:
- Агенты — **живой вызов модели** прямо в дашборде (а не заглушки и не фоновые задачи — это на потом).
- Календарь — **реальная запись** событий, а не только чтение.

Реализация:
- Перед сборкой получил схему `create_event` (нужны `summary`, `startTime`, `endTime`; таймзону и цвет передаю отдельно) — чтобы кнопка сразу строилась под настоящий формат.
- Агенты собирают контекст из текущих задач своего домена и просят модель вернуть строгий JSON-массив из 3 задач; парсер устойчив к «болтливому» ответу (вынимает `[...]`, иначе бьёт по строкам).
- Запись в календарь: наивный ISO без оффсета + явная таймзона `Europe/Moscow`, чтобы время не «уезжало».

Осознанно отрезано в этой версии: аккаунты/шаринг, фоновые агенты, аналитика, мобильный приоритет.

Логику и данные проверяем здесь (живой артефакт), красивый UI собираем в Lovable по промпту из `docs/spec-command-desk.md`. Следующий шаг — свести: либо перенести стиль из Lovable в прототип, либо подключить дизайн Lovable к реальным данным.

### Что усвоила про сам цикл
1. Сначала артефакт намерения, потом промпт.
2. Перед кодом — прощупать реальное API, строить под факт.
3. Развороты концепции — это норма; фиксируй развилки явным выбором.
4. Режь область осознанно и записывай, что отрезано и почему.
